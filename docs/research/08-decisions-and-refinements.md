# Architecture Decisions & Refinements

> Decisions made during design review, 2026-03-26

---

## Decision 1: L3 Confidence — Per-Rule Configurable

Each rule declares its own confidence threshold for heuristic decisions.

```csharp
public interface IRule
{
    // Existing...
    RuleSchema Describe();

    // Each heuristic node in the decision tree declares:
    HeuristicPolicy HeuristicPolicy { get; }
}

public record HeuristicPolicy(
    double ActThreshold,       // >= this confidence: act silently (e.g., 0.95)
    double SuggestThreshold,   // >= this confidence: act but flag in output (e.g., 0.80)
    // Below SuggestThreshold: always ask
    bool AllowOverrideInConfig // Can project config change these thresholds?
);
```

**In practice:**

| Confidence | Behavior | Output |
|------------|----------|--------|
| >= `ActThreshold` | Execute, no flag | `"status": "completed"` |
| >= `SuggestThreshold` | Execute, flag as inferred | `"status": "completed", "inferences": [{"decision": "...", "confidence": 0.87, "rationale": "..."}]` |
| < `SuggestThreshold` | Don't execute, return question | `"status": "needs_input", "blocked_on": {...}` |

Project config can override per-rule:

```json
{
  "rules": {
    "api-endpoint": {
      "heuristics": {
        "actThreshold": 0.99,
        "suggestThreshold": 0.90
      }
    }
  }
}
```

A cautious project sets high thresholds (ask more). A mature project with strong conventions sets lower thresholds (act more).

## Decision 2: Learning From Answers — Three-Layer Feedback Loop

When a user or model resolves an L4 question, the learning follows a priority order:

### Layer A: Refine the Script (Best)

If a pattern emerges that is always true for this project, bake it into the script/rule itself.

**Example:** "CreateXxxRequest always mirrors the entity minus Id and audit fields" → this becomes a heuristic in the rule's decision tree:

```csharp
// Before: L4 — always asks
// After: L3 — heuristic with high confidence
if (entityType != null)
{
    var properties = entityType.Properties
        .Where(p => !IsAutoGenerated(p) && !IsAuditField(p) && !IsSensitive(p));
    return new HeuristicResult(properties, confidence: 0.92,
        rationale: "Derived from entity, excluding auto-generated/audit/sensitive fields");
}
```

The rule's heuristic policy (Decision 1) then determines whether this acts silently or flags.

**When to refine the script:** The pattern is structural (follows from the data shape), applies broadly across the project, and can be expressed as a deterministic filter or transformation.

### Layer B: Skill-Level Memories (Good)

When the pattern can't be expressed as code but is still useful guidance, attach it as a memory/pitfall to the skill. Scripts can query these at decision points.

```
.skill-to-cs/
├── rules/
│   └── api-endpoint.cs
├── knowledge/                       # Skill-level memories
│   ├── api-endpoint.pitfalls.json
│   └── api-endpoint.patterns.json
```

```json
// api-endpoint.pitfalls.json
{
  "pitfalls": [
    {
      "id": "billing-dto-exception",
      "context": "rootPath starts with /api/billing",
      "learning": "Billing DTOs include a CurrencyCode field even when the entity doesn't have one — it's derived from the tenant config",
      "added": "2026-03-28",
      "source": "human-feedback"
    }
  ],
  "patterns": [
    {
      "id": "search-endpoints-are-post",
      "context": "path contains 'search'",
      "learning": "Search endpoints in this project use POST with a request body, not GET with query params",
      "added": "2026-03-27",
      "source": "human-feedback"
    }
  ]
}
```

Scripts consult these at L3 decision points:

```csharp
// In the rule's decision tree
var pitfalls = LoadPitfalls("api-endpoint");
var applicable = pitfalls.Where(p => p.MatchesContext(currentParams));

if (applicable.Any())
{
    // Adjust behaviour or flag to user
    foreach (var pitfall in applicable)
        result.AddWarning($"Known pitfall: {pitfall.Learning}");
}
```

**When to use memories:** The pattern is project-specific, context-dependent, or can't be expressed as a simple code filter. It's advice, not logic.

### Layer C: Human Feedback Loop (Essential)

When a script produces wrong output or a heuristic gets it wrong, the human flags the error. This feedback must be:

1. **Captured** — structured record of what happened and what was wrong
2. **Proposed** — tool suggests whether this should become a script refinement (Layer A) or a memory (Layer B)
3. **Agreed** — operator confirms the categorisation and the specific learning

```
skill-to-cs feedback api-endpoint \
  --instance "/api/billing:POST:/" \
  --issue "Generated CreateBillingRequest without CurrencyCode" \
  --correction "Billing DTOs need CurrencyCode derived from tenant" \
  --scope project
```

The tool then proposes:

```
Feedback received. Suggested action:

  1. [Memory] Add pitfall: "Billing DTOs include CurrencyCode"
     Scope: rootPath starts with /api/billing

  2. [Script refinement] Not feasible — CurrencyCode is domain-specific,
     can't be derived from entity schema alone.

  Apply option 1? [Y/n]
```

The operator confirms. The pitfall is saved. Next time the rule encounters a billing DTO, it flags: "Known pitfall: Billing DTOs need CurrencyCode."

**The feedback → refinement pipeline:**

```
Error occurs
  → Human flags it (skill-to-cs feedback)
  → Tool classifies: script-refineable or memory-only?
  → Tool proposes specific learning
  → Operator agrees/adjusts
  → Learning persists (script commit or knowledge file)
  → Next invocation uses the learning
```

### When Learnings Promote

Over time, memories can graduate to script refinements:

```
Memory (Layer B): "Billing DTOs always include CurrencyCode"
  ↓ After 5 billing endpoints all needed it
  ↓ Tool suggests: "This pattern has been consistent. Promote to script heuristic?"
  ↓ Operator agrees
Script (Layer A): Rule now auto-adds CurrencyCode for /api/billing DTOs
```

And script heuristics can promote to hard rules:

```
Heuristic (L3, 0.92 confidence): "Exclude audit fields from request DTOs"
  ↓ After 20 endpoints with zero corrections
  ↓ Tool suggests: "This heuristic has never been wrong. Promote to L2 (always apply)?"
  ↓ Operator agrees
Decision tree (L2): Always exclude audit fields, no confidence check
```

## Decision 3: Model Integration — Bubble to Caller (V1), Direct Call (V2 Option)

**V1:** Always bubble. Return structured questions. The calling agent/skill/human provides judgment. Keeps the tool decoupled and testable.

**V2 option:** Optionally call a small model for L4 decisions. Configurable per-rule, off by default. When enabled, the tool resolves more autonomously.

## Decision 4: Sub-Script Granularity — Atomicity Primary, Layered for Reuse and Clarity

### The Sizing Rule

**Each leaf script does exactly one atomic action: a single add, update, or delete on a single target.**

The decision tree above the leaves is layered for two goals simultaneously:
- **Reuse** — shared sub-scripts that multiple rules invoke
- **Clarity** — a developer reading the tree can follow the logic

### Atomic Actions (Leaf Nodes)

| Action | Granularity | Example |
|--------|------------|---------|
| **Add line to file** | One insertion in one file | Add `MapGet(...)` line |
| **Add file** | One new file | Create `CreateUserRequest.cs` |
| **Update section** | One contiguous region in one file | Replace a method body |
| **Delete line/block** | One removal from one file | Remove a mapping registration |

A leaf never touches two files. A leaf never makes two unrelated changes to one file.

### Decision Nodes (Branch Nodes)

Decision nodes don't write — they resolve a question and route to the next node:

| Node | Reuse Potential | Example |
|------|----------------|---------|
| `resolve-file` | High — every rule that targets a file needs this | "Given a convention path, find or create the target file" |
| `resolve-insertion-point` | High — any rule inserting into an existing file | "Find the right line in this file" |
| `check-type-exists` | High — any rule that references or creates types | "Does UserDto exist? Where?" |
| `apply-convention` | Medium — project-specific naming/layout | "What's the models directory?" |
| `evaluate-heuristic` | Medium — shared heuristic patterns | "Derive DTO properties from entity" |

### The Layering

```
Rule (orchestrator — L3)
├── Shared decision nodes (L2/L3 — reused across rules)
│   ├── resolve-file
│   ├── resolve-insertion-point
│   ├── check-type-exists
│   └── apply-convention
├── Rule-specific decision nodes (L2/L3 — unique to this rule)
│   ├── determine-method-group
│   └── derive-result-type
└── Atomic action leaves (L1)
    ├── insert-line
    ├── create-file
    └── append-to-section
```

Shared nodes live in the engine. Rule-specific nodes live in the rule. Leaves are generic utilities.

### Implementation Shape

This isn't separate `.cs` files per node — it's a method-level decomposition within the rule, with shared utilities extracted to the engine:

```csharp
// Engine-level shared utilities
public class FileResolver { ... }         // resolve-file
public class InsertionPointFinder { ... } // resolve-insertion-point
public class TypeChecker { ... }          // check-type-exists
public class ConventionResolver { ... }   // apply-convention
public class AtomicWriter { ... }         // insert-line, create-file, etc.

// Rule-level methods
public class ApiEndpointRule : IRule
{
    // Decision nodes (call shared utilities + rule-specific logic)
    private EndpointFile ResolveEndpointFile(RuleParams p, ProjectContext ctx) { ... }
    private InsertionPoint FindMethodGroup(EndpointFile file, string method) { ... }
    private ResultType DeriveResultType(string[] errorCodes) { ... }

    // Orchestrator
    public async Task<GenerationResult> GenerateAsync(RuleParams p, ProjectContext ctx, CancellationToken ct)
    {
        var file = ResolveEndpointFile(p, ctx);           // L2 shared
        var point = FindMethodGroup(file, p.Method);       // L2 rule-specific
        var resultType = DeriveResultType(p.ErrorCodes);   // L2 rule-specific

        var requestModel = await EnsureModel(              // L2/L3/L4 with bubble
            p.RequestModel, p, ctx, ct);
        if (requestModel.NeedsInput)
            return GenerationResult.NeedsInput(requestModel.Question);

        // All resolved — pure L1 from here
        var fragments = new List<CodeFragment>
        {
            AtomicWriter.InsertLine(file, point, RenderMapping(p, resultType)),
            // ... more atomic actions
        };

        return GenerationResult.Success(fragments);
    }
}
```

When a rule gets complex enough that its decision tree warrants its own file, it graduates to a sub-rule — another `IRule` that the parent composes. This is the rules-calling-rules composition from the earlier design.

## How These Decisions Change the V1 Plan

| Phase | Change |
|-------|--------|
| **Phase 2 (Parameters)** | Add `HeuristicPolicy` to schema. Add per-rule config overrides for thresholds. |
| **Phase 4 (CodeFragmentWriter)** | Rename to `AtomicWriter`. Enforce one-file-per-leaf. Add transactional multi-fragment coordination. |
| **Phase 5-6 (Rules)** | Each rule documents its decision tree with layer classifications. Heuristic nodes return confidence. Bubble-back returns structured questions. |
| **Phase 8 (Interactive)** | Add `feedback` command. Add `knowledge/` directory with pitfalls and patterns. |
| **Phase 9 (Polish)** | Add promotion suggestions (memory → script, heuristic → hard rule) based on track record. |
| **New: Phase 8.5** | Feedback loop — `skill-to-cs feedback` command, learning classification, operator confirmation. |

## Summary of Architectural Principles

1. **Per-rule confidence thresholds** — each rule knows its own reliability at each decision point
2. **Three-layer learning** — refine script > add memory > capture feedback, in that priority order
3. **Operator agreement required** — no learning persists without human confirmation
4. **Promotion over time** — memories graduate to heuristics, heuristics graduate to hard rules, tracked by consistency record
5. **Atomicity at the leaf** — one action, one file, one target
6. **Shared decision nodes** — engine provides reusable resolution utilities
7. **Clarity and reuse together** — layer depth serves both readability and composition
8. **Bubble, don't guess** — V1 never fills in what it can't determine; it returns the question
